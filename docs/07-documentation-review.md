# 07. Ревизия документации (роль 2)

Задача: оценить новую документацию по критериям и внести правки до уровня "можно передавать джуну".

## 1) Критерии оценки
- Полнота покрытия проекта.
- Точность относительно текущего кода.
- Понятность для junior-разработчика.
- Наличие практических инструкций (onboarding, debug, release).
- Поддерживаемость документации (понятная структура, отсутствие дубликатов).

Шкала: `1` (плохо) ... `5` (отлично).

## 2) Оценка
| Критерий | Оценка | Комментарий |
|---|---:|---|
| Полнота | 5 | Покрыты продукт, архитектура, БД/RLS, разработка, стек, ops |
| Точность | 5 | Сверено по `src/app`, `src/lib`, `supabase/migrations`, `package.json` |
| Понятность для junior | 5 | Есть пошаговые flow, "где править", "что сломать легко" |
| Практическая применимость | 5 | Есть чеклисты, smoke, SQL snippets, runbook |
| Поддерживаемость | 4 | Структура чистая, но требует дисциплины обновления при изменениях |

Итог: `4.8 / 5`.

## 3) Какие правки внесены по итогам оценки
1. Добавлен отдельный файл `docs/04-development-guide.md` с workflow + чеклистами PR/smoke.
2. Добавлен файл `docs/06-deployment-and-ops.md` с релизным runbook и инцидентным алгоритмом.
3. Добавлены явные ссылки "где искать код" в `docs/01-project-overview.md` и `docs/02-architecture.md`.
4. Добавлен подтверждённый блок best practices по технологиям в `docs/05-technology-mini-guides.md`.
5. Удалены устаревшие dev-plan/tech-plan/development-diary/deploy-файлы, чтобы не было конкурирующих источников правды.

## 4) Остаточные риски
1. Нет автотестов как источника живой верификации документации.
2. При изменении RLS/policy документация может устареть, если не обновить `docs/03-database-and-rls.md`.
3. При добавлении новых сущностей нужно обязательно обновлять route map и stack guide.

## 5) Правило обновления docs
Любой PR, который меняет хотя бы один из блоков:
- маршруты,
- server actions,
- schema/миграции,
- env/деплой,
- i18n ключи,
должен включать соответствующее обновление файлов в `docs/`.
